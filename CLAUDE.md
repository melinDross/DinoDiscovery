# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

AI-powered dinosaur generator where children pick attributes (size, habitat, diet, feature, personality) via a wizard, get an AI-generated dino (image + name + description), and receive a collectible "card" reframed as an ARG (alternate-reality-game) species discovery — deterministic species ID, rarity tier, illustrated habitat art, 3D flip/tilt interaction.

Live: https://dino-discovery-generator.pages.dev

Tech stack: Vite + React 19 + TypeScript (strict) + Tailwind CSS, deployed on Cloudflare Pages with Wrangler (Pages Functions for the backend, KV for caching/rate-limiting/results, R2 for generated images).

## Commands

```bash
npm install
npm run dev        # Vite dev server (frontend only, no backend functions)
npm run build       # tsc --noEmit && vite build
npm run preview     # preview the production build locally
npm test            # vitest run — full suite
npm test -- path/to/file.test.ts          # single test file
npm test -- path/to/file.test.ts -t "name" # single test by name
npm run typecheck   # tsc --noEmit only
```

Local full-stack dev (frontend + Pages Functions): copy `.dev.vars.example` to `.dev.vars`, fill in API keys, then either:
- `npx wrangler pages dev dist --compatibility-date=2026-06-28 --compatibility-flags=nodejs_compat` (after `npm run build`), or
- `npx wrangler pages dev -- npm run dev` for hot-reloading the frontend while functions run.

One-off asset generation scripts (NOT part of the build, run manually, need `OPENAI_API_KEY` exported in the shell — `set -a && source .dev.vars && set +a` works):
```bash
node scripts/generate-landing-logo.mjs          # public/landing-logo.png
node scripts/generate-medallion-icons.mjs       # public/icons/medallions/*.png (27 files)
node scripts/remove-medallion-backgrounds.mjs   # post-process: adds real alpha transparency (no API key needed)

```

### Deploy

1. `npx wrangler kv namespace create RATE_LIMIT_KV` / `CACHE_KV` / `RESULTS_KV`, copy IDs into `wrangler.jsonc`
2. `npx wrangler r2 bucket create dino-discovery-images`
3. `npx wrangler pages project create dino-discovery-generator --production-branch main`
4. `npx wrangler pages secret put ANTHROPIC_API_KEY / OPENAI_API_KEY / KIT_API_KEY / KIT_FORM_ID / ADMIN_KEY --project-name dino-discovery-generator`
5. `npm run build && npx wrangler pages deploy dist --project-name dino-discovery-generator`

## Architecture

### Request flow (live generation)

`App.tsx` drives a single-page state machine (`'landing' | 'wizard' | 'loading' | 'result' | 'error'`). The wizard collects `DinoAttributes` (size, habitat, diet, feature, personality) one screen at a time via `AttributeGroup`, auto-advancing ~500ms after each pick (`scheduleAdvance`/`clearPendingAdvance` in `App.tsx`, guards against double-submission on rapid clicks). **Each option's icon is the same medallion image used on the finished card** (`ATTRIBUTE_MEDALLION_PATHS` from `cardTheme.ts`), not a separate emoji set — `data/attributes.ts`'s old `ATTRIBUTE_EMOJIS` map was deleted when this switched over, so the wizard and the card now visually preview the same icon instead of a mismatched emoji standing in for it.

**The wizard's attribute options are mini-cards, not circular/rounded-square icon buttons** — two earlier iterations both got the same "hard to read/too small" feedback: first a small circular crop (hid the artwork's corners), then a bigger rounded-square icon-plus-label pill. The current design (`AttributeGroup.tsx`) is a small vertical card per option: the medallion image fills the full tile width as a fixed-height band at the top (`w-full h-[108px]`/`h-[132px]`, matching the tile's own width, `w-[108px]`/`w-[132px]`), and the option name sits in a caption strip below it (`flex-1 flex items-center justify-center`, so the strip's height adapts to wrapped two-line labels). The button itself carries `overflow-hidden rounded-2xl` so the image's top corners get clipped to match the tile shape without the image needing its own border-radius. This reads clearly at both a 6-across desktop row and a 3×2 mobile wrap (verified visually via Playwright at both widths) — don't shrink the tile width much further, six of them barely fit a typical desktop viewport as-is. **The image band's height is deliberately equal to the tile's width, not shorter** — since the band uses `object-contain` on a square medallion image, a square band lets the circle fill it completely; an earlier version had a shorter, non-square band (band narrower than tile width), which under `object-contain` sized the circle down to fit the smaller dimension and left dead space on the sides — bigger requests should grow both the width and height together to stay square, not just one.

**Use `object-contain` on that image band, not `object-cover`** — the medallion art is a 1:1 square (a circle touching all four edges of its canvas), but the band itself is wider than tall (e.g. 104×84), so `object-cover` (tried first) crops the top/bottom of the circle to fill the non-square band, visibly cutting the medallion off. `object-contain` shows the full circle with empty space left/right instead, which is only viable now that the medallions have real alpha transparency (see `remove-medallion-backgrounds.mjs` below) — before that fix, `object-contain`'s letterboxing would have shown the medallion's own square opaque background instead of blending into the tile.

`Card.tsx`'s own medallions (inside the finished card's compact attribute panel, where space is much tighter) still use a small circular crop (`rounded-[40px]`) — that's a deliberate difference for a space-constrained context, not an inconsistency to "fix" to match the wizard. If circular medallions are ever wanted again anywhere, remember `rounded-full` in this project's Tailwind theme resolves to `4px` (a deliberate near-zero-radius look used elsewhere), not an actual circle — use an arbitrary value like `rounded-[999px]` instead.

On the last pick, `handleDiscover` POSTs to `/api/generate-dino` (`functions/api/generate-dino.ts`):
1. Rate-limits by IP (`functions/lib/rateLimit.ts`: 5/hour, fixed window in KV) unless `X-Admin-Key` matches `ADMIN_KEY` (`functions/lib/adminAuth.ts`, constant-time compare).
2. Computes a cache key from the 5 attributes (`functions/lib/hash.ts`: SHA-256 of `size|habitat|diet|feature|personality|v2` — the `v2` suffix exists to invalidate old cache entries when the generation prompt changed background color; bump it again if the image/text prompts change in a way that should bust the cache).
3. On cache hit (`functions/lib/cache.ts`, KV, no TTL — cached forever): reuses the stored text + image key, still creates a **new** `resultId` and result record (so the same attribute combo can be "discovered" by multiple kids with separate shareable links/discoverer names, but a single generated image/text asset).
4. On cache miss: calls Anthropic (`functions/lib/anthropic.ts`, `claude-haiku-4-5`, strict JSON-only prompt for `{scientificName, commonName, description}`) and OpenAI (`functions/lib/openai.ts`, `gpt-image-2`, quality `low`, **prompt pins the background to a solid `#0d1a0f`** — see "Dino cutout" below, this is load-bearing) in parallel, stores the image in R2 (`functions/lib/r2.ts`), caches the text+imageKey, then proceeds like a hit.
5. Saves a `ResultRecord` (`functions/lib/results.ts`, KV, 90-day TTL) keyed by `resultId`, including the full `attrs` — needed so `/r/:id` shared links can rebuild the card client-side, not just show the image.

Images are served via `functions/images/[key].ts` (R2 → `image/png`, immutable 1-year cache).

`/r/:id` deep links: `App.tsx`'s `loadResultFromUrl` calls `GET /api/results/:id` (`functions/api/results/[id].ts`), which returns the stored record **including `attrs`** — both the live-generation path and the shared-link path converge on a single `currentAttrs` state in `App.tsx`, so the result screen never renders without attrs (gate: `flowState === 'result' && result && currentAttrs`).

### Email capture (Kit)

`EmailGateModal` → `subscribeEmail` → `POST /api/subscribe` (`functions/api/subscribe.ts`) → `functions/lib/kit.ts`'s two-step Kit v4 flow (create subscriber, then add to form). This is explicitly best-effort and **never blocks the download** — `App.tsx`'s `handleEmailConfirm` calls `subscribeEmail` in a try/catch that swallows failures, then always calls `downloadCertificate()` regardless of outcome.

### Landing page (`src/components/Landing.tsx`)

A looping-once video (`public/dino-landing.mp4`, the dino bursting through a cracked surface) replaces what was originally a static logo image, wrapped in a `<button>` (not a `<div>`) so it's properly tappable/keyboard-accessible. **The `muted` JSX prop on `<video>` doesn't reliably make it to the actual DOM attribute** in all React versions/timings — and iOS Safari refuses to autoplay a video that isn't muted *at the DOM level*, not just as a React prop — so `muted` is also set imperatively (`el.muted = true`) inside the ref callback, alongside an explicit `el.play()` call (both guarded for environments where `play()` doesn't exist or doesn't return a Promise, e.g. jsdom in tests). The video has no `loop` — it plays once on mount and holds its last frame. Tapping the button replays the video from `currentTime = 0` **and** plays `public/t-rex-sound.mp3` from `ROAR_START_SECONDS = 8` (skipping the file's lead-in) to its end, together — this is also why the roar can't just autoplay alongside the video: browsers require a direct user gesture for audio playback, full stop, so it's tied to the tap handler, not any video lifecycle event.

**Wordmark** (`public/dino-discovery-logo.png`): originally a cropped, opaque rectangular slice of `card-back.png`'s logo art needing a `mask-image` radial-gradient fade to blend its edges into the page background (see prior revisions of this doc/file history if that technique is ever needed again elsewhere). **The user has since supplied a redrawn version of this same logo with a real alpha channel** — no mask is needed anymore, and the `<img>` renders at its natural aspect ratio with no cropping artifacts to hide.

**Video/logo overlap composition**: the wordmark is laid out as a *second*, later sibling of the video's wrapping `<div>` (not nested inside it), pulled upward with a negative top margin (`-mt-[73px]`/`[101px]`/`[112px]`, responsive per breakpoint) and given a higher `z-index` than the video, so the logo renders **in front of** the video and covers roughly the bottom 35% of it. This reads as the dino bursting up out from behind the wordmark rather than two independently stacked images with a gap — the negative-margin amount is a fixed px value per breakpoint (not a Tailwind fraction of the logo's own box) because it's tuned as a percentage of the *video's* rendered height, which scales with the separate `w-52`/`w-72`/`w-80` classes on the video's own wrapper, not the logo's `max-w-*` classes. If either element's breakpoint widths change, recompute the overlap constants (video height × 0.35) rather than assuming the logo's width scale keeps them in proportion.

**Progressive blur at the video's base**: a small absolutely-positioned `div` sits inside the video's `overflow-hidden` wrapper (as a sibling *after* the `<video>` tag, so it draws on top), spanning the bottom 45% of the video's box, with `backdrop-blur-md` and a `mask-image` linear-gradient fading its own opacity from transparent (top) to solid (bottom). This makes the blur strengthen gradually toward the bottom edge instead of snapping on at a hard line, so the seam where the opaque logo now overlaps the video reads as a soft transition rather than a visible cut. This blur is deliberately scoped to just the video element — the faux card-silhouette decorations behind it (see below) are untouched by it.

**Faux rarity-card silhouettes**: two CSS-only tilted rounded-rect `div`s (rotated ±11-14°, rainbow-gradient fills, no image) sit behind the video inside the same relative-positioned wrapper, peeking out at the corners to hint at the card-collectible payoff before the wizard even starts. Not the real `Card` component — kept intentionally lightweight since this is landing-page flavor, not a functional preview.

**Caption + live discovery counter**: below the logo, two lines — a fixed tagline ("Cada carta es una especie única") and a counter (`#XXX,XXX descubiertas`, `formatDiscoveryCount` pads to 6 digits and inserts a comma at a fixed position, deliberately not `toLocaleString` which wouldn't zero-pad). **This is a flavor stat, not real data** — it starts at a hardcoded base and ticks up by 1 every `DISCOVERY_COUNT_INTERVAL_MS` (4s) via a `setInterval` in a `useEffect`, purely to feel like a living ARG counter while the landing screen is open. It resets on every page load/reload — there's no persistence (localStorage/backend) backing it, so don't mistake it for a real discovery tally if this is ever wired to actual analytics later.

**Brand palette for buttons and UI chrome** (`tailwind.config.js`'s `brand` color + `display2` font family): restyled to match this logo — magenta/pink (`brand: '#ec1a8f'`, picked by eye from the logo, not extracted programmatically) plus the `Fredoka` Google Font (`display2`) instead of the app's existing lime-green `accent` (`#b2ff00`) + `Bangers` (`display`). This covers every button project-wide (landing, wizard picks, result screen, card scene, name step, email gate, error retry) **and** the surrounding UI chrome outside the generated card — headings (`AttributeGroup`, `NameStep`, `EmailGateModal`), the wizard progress bar, the `LoadingDino` caption/progress bar, and form input borders/focus rings. **The generated dino card itself is deliberately excluded** from this brand pass — its border, medallion panel, rarity badge, and star ratings still don't follow the `brand`/`display2` tokens; a full rebrand would also touch many hardcoded color literals scattered outside the token system (card frame/panel colors, medallion panel gradients, etc.) that don't automatically follow a token rename. If the card is ever brought onto the `brand` token system, audit those hardcoded literals separately — swapping just the token wouldn't reach them. (The card's own palette *has* since changed independently — see "Card layout" below — just not onto the `brand`/`display2` tokens used elsewhere.)

### Page background (`grid-overlay` in `src/index.css`)

Originally a flat dark-green tech-grid look; redesigned to a "fantasy night sky" — deep indigo base (`#07091a`) with colored radial glows (emerald top-left, violet bottom-right, gold upper-right) and a sparser/dimmer star-grid, applied via `html`/`body` background-color too (not just the `.grid-overlay` class) so iOS overscroll-bounce doesn't reveal a different color underneath. The card's `.card-perspective` carries its own `filter: drop-shadow(...)` (dark contact shadow + green accent bloom) specifically so it still reads as floating above this much busier background instead of blending into it — if the background palette changes again, sanity-check that contrast.

### The card (ARG layer)

This is the most-iterated part of the codebase; read this before touching `Card.tsx`/`CardScene.tsx`.

**Rarity system** (`src/utils/speciesHash.ts`, spec in `docs/RARITY_SYSTEM.md`): each of the 5 attributes has a point table; `calculateRarityScore` sums them and applies special-combo multipliers (e.g. Coloso+Oófago+Alas ×1.5), `calculateRarity` buckets the total into 5 tiers (common/uncommon/rare/epic/legendary, thresholds 9/12/15/18), `RARITY_TIER_NUMBERS` maps tiers to 1-5 for the "Tier" display and star count. **`'Gigante'` intentionally carries the point value `docs/RARITY_SYSTEM.md` assigns to `'Gigantesco'`** — it was *not* renamed to avoid breaking the many existing tests/UI strings that already say `'Gigante'`; this is documented inline in the point table, don't "fix" it without checking that comment first.

`generateSpeciesId` hashes the sorted attribute values into a deterministic `DX-XXX-XXX` code — same attrs always produce the same ID, reinforcing the "this species already existed" narrative.

**Habitat art** (`src/data/cardTheme.ts`): each of the 6 habitats has 2 illustrated sub-biome backgrounds (`public/habitats/{slug}-1.png` / `-2.png`, real AI-illustrated assets, not placeholders). `pickHabitatBackground(attrs)` selects one deterministically (char-code sum of all 5 attribute values, mod 2) — same attrs always render the same scene.

**Dino cutout** (`src/utils/dinoCutout.ts`): the generated dino image has a *flat, opaque* background (no alpha channel — OpenAI doesn't return transparent PNGs from this prompt), but it's pinned to the known solid color `#0d1a0f` by the generation prompt in `functions/lib/openai.ts`. `cutoutDinoImage` chroma-keys that exact color out client-side (canvas `getImageData`, distance-threshold + feather), so the dino composites as a true cutout over the habitat art instead of showing as a flat rectangle. **This is fragile**: if the OpenAI prompt's background color or the cache-key version changes, `BACKGROUND_COLOR` in `dinoCutout.ts` must be updated to match, or cutout silently fails (no error — `useDinoCutout` in `Card.tsx` falls back to the original, un-cut image on any load/decode failure, so the failure mode is "looks like before this feature existed," not a crash).

**Card visual style — "Terminal ARG"**: the card's interior palette/typography is a near-black glass panel (`#050b12`/`#0a1420`), cyan/magenta radial-gradient glow, and monospace type (`font-mono`) throughout the panel/badges/name/description — replacing an earlier green/purple fantasy palette (`Bangers`/`font-display`, lime `accent` glows). Chosen from a 4-way live-preview comparison (a dev-only `?variants` route + `CARD_STYLE_TOKENS` indirection, both since deleted — this is now hardcoded directly as named constants at the top of `Card.tsx`, e.g. `PANEL_BACKGROUND`, `TEXT_COLOR`) against a parchment/museum-plate look and a comic/sticker look, to lean into the "alternate-reality-game species discovery" framing more explicitly. If another restyle is ever wanted, the same throwaway-preview-route technique is a fast way to compare options live before committing to one — don't restyle `Card.tsx` blind.

**The outer frame ring is a separate warm amber gradient** (`FRAME_BORDER_COLOR`/`FRAME_BG`/`FRAME_TEXTURE` in `Card.tsx`), not the cyan glass tone used for the interior panel — picked from a live comparison against cyan/indigo/emerald borders for warmth against the cool interior glow. The frame is a **two-layer structure**: an outer `div` (`p-[20px] rounded-[28px]`, `backgroundImage: FRAME_TEXTURE`) wrapping an inner `div` (`rounded-[10px]`, `backgroundColor: FRAME_BG`) that holds the habitat art/panel/footer — this replaced a single `div` with a plain CSS `border` once the border itself needed a gradient fill, since a CSS `border-color` can't take a gradient (`border-image` could, but doesn't support box-shadow/rounded-corner combinations as cleanly as this padding-based approach). Two CSS-only frame textures (a cross-hatched "hammered" look and a diamond "scaled" mesh, both via layered `repeating-linear-gradient`/`repeating-radial-gradient`) were also tried live and dropped — both read as busy/noisy at the ring's 20px width; the flat gradient (`FRAME_TEXTURE`) reads cleaner. **The card's corners must stay rounded** (`rounded-[28px]` outer / `rounded-[10px]` inner) — an earlier pass of the neon restyle briefly squared them off (`rounded-[4px]`) along with the smaller badges/medallions, which was reverted as an explicit fix; the small interior badges (species-ID pill, rarity badge, medallion images) keeping their own sharper `rounded-[4px]` corners is intentional and separate from the outer frame's rounding.

**Card layout** (`src/components/Card.tsx`): stacked in three sibling layers, intentionally *not* nested, so z-index ordering is simple:
1. The framed box (rounded corners, `border-[20px]`, `overflow-hidden`) — habitat background, stone attribute-medallion panel, score/rarity/tier row, footer. Anything that must stay clipped to the card's rounded shape goes here.
2. The dino `<img>` (cutout, `scale-105`, `object-top`, multi-layer `drop-shadow` filter, `z-20`) — rendered as a **sibling after** the framed box, specifically *not* inside its `overflow-hidden`, so a big creature's claw/tail/horn can visually pop past the black border for a 3D effect. Positioned with inline styles using `CARD_BORDER_PX`/`ART_HEIGHT_PX` constants that must stay in sync with the framed box's `border-[20px]` / `h-[440px]` Tailwind classes (duplicated because Tailwind arbitrary-value classes can't be parameterized from JS at build time). The scale was originally `110%` centered (`object-contain`'s default), but real-device testing showed dinos getting visually cropped at the bottom edge near the medallion panel — pulled back to `105%` and switched to `object-top` so any letterboxing slack from the source image's aspect ratio collects at the bottom (away from the panel) instead of being split top/bottom.
3. A text/tag overlay (species ID, rarity badge, sub-biome name, name/sci-name/description, `z-30`) — a **further sibling after** the dino layer, so text stays legible no matter how far the dino's cutout extends. (Earlier iteration had the dino on top of everything including this text — don't regress to that.)

**The attribute medallion `<img>`s need `max-w-none`** — Tailwind's preflight sets `img { max-width: 100% }`, which silently clamped the larger center "Poder" medallion (`w-16` = 64px) down to its grid column's actual width (~59px on most cards, narrower than 64px once `gap-1` and the 5-column split are accounted for). The result was a non-square image box that the `rounded-[40px]` clip then rendered as a flat-topped shape instead of a circle — easy to miss because the other four medallions (`w-12` = 48px) fit within the column without being clamped, so only the center one looked broken. Any future medallion/icon sizing in this component should include `max-w-none` if the element is wider than its grid cell.

**The gradient overlay "clips" medallions if anything pokes above the panel's content area** — there are actually two separate causes, each of which created the same visual (medallion tops apparently cut off), both from the text/tag overlay's description gradient (`z-30`, `bg-gradient-to-t from-bg`, anchored `bottom-0` within the 440px art area) being fully opaque at its own bottom edge, which is exactly where the panel starts:

1. *Panel had a negative top margin (-mt-[10px])* — pulled the whole panel up into the gradient's opaque zone. Fixed by removing that margin.
2. *Center cell has -mt-7 (28px) pop-up* — even after removing the panel's own margin, the center cell's -mt-7 pulled its icon 12px above the panel's top boundary (since pt-4=16 < 28), back into the gradient zone. Fixed by increasing the panel's `pt-4` to `pt-8` (32px), so the icon lands at pt-8(32) - mt-7(28) = +4px inside the panel — safely below the gradient.

The `-mt-7` pop-up on the center cell is intentional design (makes the "PODER" icon visually dominant / elevated). The `pt-8` on the panel is what makes it work without the gradient painting over the icon top. Do not reduce `pt-8` below 8 (32px) without also reducing or removing `-mt-7`. Don't reintroduce any `-mt-X` on the panel itself (pulls everything up, not just the center icon). The panel's background is also no longer a flat `#1c1c1c` — it's a layered `radial-gradient`/`linear-gradient` (green + purple accents) to read as fantasy-themed rather than a flat dark box, generated with CSS only (no new AI-illustrated asset, to keep scope small — if a richer illustrated texture is wanted later, generate one the same way `scripts/generate-card-back.mjs` does and swap this gradient for a background-image).

**The per-attribute category label (`"TAMAÑO"`/`"DIETA"`/`"PODER"`/etc.) was removed from each medallion cell** — with the colorful, visually distinct medallion icons (see below) plus the value text, the category label was judged redundant. `cell.label` still exists in `Card.tsx`'s `cells` array (used for the `key` prop and the `alt` text, e.g. `"Tamaño: Coloso"`) even though it's no longer rendered — don't treat that as dead code to delete. Removing the label line freed up vertical space, used to size the medallions up: `w-12 h-12`/`w-16 h-16` (48px/64px) became `w-14 h-14`/`w-[72px]` (56px/72px). This still fits within the panel's available width at the card's fixed 420px design width — verified visually via Playwright screenshot — but don't push the sizes up further without re-checking, the 5-column row was already close to the panel's content width at these sizes.

`ATTRIBUTE_MEDALLION_PATHS` (`cardTheme.ts`) maps every attribute *value* (26 total) to a generated icon at `public/icons/medallions/{slug}.png`, plus a `emblem.png` brand mark. All 27 were generated once via `scripts/generate-medallion-icons.mjs` and committed as binary assets — there's no runtime generation. **Regenerated once already** to move off a uniform dark-grey stone bas-relief look (monotone, felt at odds with the app's more vibrant branding) to a glossy enamel/gemstone style with a distinct accent color *per attribute category* — size: amber-orange, diet: green-teal, feature: hot pink-magenta, personality: purple-violet, habitat: sky blue-cyan, emblem: rainbow (the odd one out, deliberately) — so the five medallion rows on a card read as visually distinct groups instead of a monotone set. The script's `ICONS` array now carries an `accent` field per entry, substituted into `STYLE_SUFFIX`'s `ACCENT_COLOR` placeholder. Re-running the script costs real OpenAI API spend (27 image generations) — don't run it speculatively.

**`scripts/remove-medallion-backgrounds.mjs`**: a second one-off post-process, run once after the regeneration above, that actually gives the medallions real alpha transparency — `gpt-image-2` doesn't reliably return true transparency even when asked (same limitation as the dino cutout photos, see `dinoCutout.ts` below), and on at least one icon (`mediano.png`) the "background" it returned was a faint two-tone checkerboard texture (the model's own visual placeholder for "transparent"), not a flat color. The script flood-fills from the image border inward using Node's `sharp` package (an explicit `devDependency` — it was already present transitively via `wrangler`/`miniflare`, but relying on an undeclared transitive dependency for a script that needs it directly was fragile), clearing any pixel connected to the border that's within a color-distance threshold of the border's own average color, stopping naturally at real color edges (the gold ring) so it doesn't eat into legitimate near-white highlights inside the artwork. Both this and the generation script are one-off/manual — run order matters (generate, then remove backgrounds) if the set is ever regenerated from scratch.

**Card back / flip** (`src/components/CardScene.tsx`): a true two-sided flip (`backface-visibility: hidden` on both `.card-face` elements, back pre-rotated 180deg in CSS), not a fake rotation of the same front content. `public/card-back.png` was AI-generated (`scripts/generate-card-back.mjs`) to approximate a reference logo image the user pasted in chat — **Claude has no tool to extract pasted/attached image bytes to disk**, so any future "use this exact image" request needs the user to save the file into the repo themselves; the fallback is generating a close approximation via the same one-off script pattern.

**The 3D flip properties need `-webkit-` prefixes** (`perspective`/`transform-style: preserve-3d`/`backface-visibility`/the back face's `rotateY(180deg)`) in `src/index.css` — iOS Safari silently ignores the unprefixed versions. **This alone was not sufficient**: real-device testing on iOS Safari still showed the front and back faces blended/corrupted together mid-flip even with prefixes — a known WebKit bug where `backface-visibility: hidden` doesn't reliably hide a face when it (or a descendant, here the dino's `drop-shadow` filter layer) is inside a 3D-transformed subtree. The actual fix is a **browser-agnostic JS fallback** in `CardScene`: a `facing: 'front' | 'back'` state explicitly toggles `visibility` on each `.card-face` element, timed via `setTimeout` to the approximate 90°/270° crossing point of each rotation (half the flip-in duration; roughly the first and third quarters of the spin duration) — independent of whether `backface-visibility` itself works. Keep both the CSS property and the JS toggle; don't remove either. This can't be verified by Playwright/Chromium (which doesn't reproduce the bug) — only by testing on a real iOS Safari device.

CSS for the 3D mechanics lives in `src/index.css` under `@layer utilities`: `.card-perspective` (flip/tilt/spin container — **no `filter` here**, a filter on the perspective container creates a stacking context that flattens 3D rendering on Safari) / `.card-scene-wrapper` (sits *outside* `.card-perspective`, carries the `filter: drop-shadow(...)` lift effect) / `.card-flipper` (flip+tilt+spin, `border-radius: 48px`) / `.card-face` / `.card-face-back`.

**Rarity visual (holo-foil sheen, `Card.tsx`)**: rare/epic/legendary cards get a holographic foil sheen (`FOIL_ELIGIBLE_RARITIES`); common/uncommon don't. This went through three failed approaches before landing here — worth reading before touching it again:
1. Animating `box-shadow` on `flipperRef` (`.card-flipper`, inside `.card-perspective`) — travelled with the card, but reintroduced the iOS habitat-flash-during-drag compositing bug (continuously mutating *any* visual property inside the 3D transform tree triggers it, not just CSS `animation`).
2. Animating `filter` on `innerRef` (`.card-scene-wrapper`, an ancestor of `.card-perspective`) — avoided the compositing bug, but silently rendered nothing (WebKit/Chromium don't reliably rasterize `filter` on an ancestor of a `preserve-3d` subtree).
3. A separate `<canvas>` glow, filled and drawn behind the card in DOM order to fake the occlusion real `box-shadow` gets for free — this worked without either bug, but the effect itself ("a glow") wasn't what was wanted; also, an extra rendering surface for a purely decorative effect was more machinery than justified once approach 4 below turned out simpler.

**Current approach**: a plain `<div>` inside `Card.tsx` itself (not `CardScene.tsx`), positioned as the last sibling in the card's stack (above the dino at `z-20` and the text overlay at `z-30`, `z-40`), with a rainbow `linear-gradient` background and a `mask-image`/`WebkitMaskImage` fade so it disappears before reaching the description text block (same region the text overlay's own gradient backing starts fading in). Its `background-position` is driven by a `foilTilt` prop (`{x, y}`, normalized -1..1) — **no new render loop**: this piggybacks on the one class of continuous update already proven safe inside the 3D scene (`transform`/tilt-driven interaction), rather than `box-shadow`/`filter`, which is what broke approaches 1 and 2. `Card`'s `foilTilt` prop defaults to `{x: 0, y: 0}` so it still renders correctly when used standalone (e.g. the hidden `html2canvas` capture node in `App.tsx`, which doesn't wire tilt through).

**`foilTilt.x` is `Math.sin(totalRotateY in radians)`, not `tilt.rotateY` directly — this was a real bug, not a style choice.** `tilt.rotateY` is only ever set by desktop mouse hover (`handleMouseMove`); touch-drag always leaves it at `0`, using `spinDeg` for rotation instead. Since dragging on an actual phone is the primary way anyone interacts with this card, driving the foil from `tilt.rotateY` meant it never reacted to rotation at all on touch devices — it only ever reflected the small vertical `tilt.rotateX`-driven parallax. `sin(totalRotateY)` reacts to the card's actual total rotation (spin, flip, everything), is naturally bounded to -1..1 without extra clamping, cycles smoothly through a full rotation, and is `0` when the card faces the viewer head-on (0°/360°) so it doesn't fight the resting diagonal-band offset baked into `Card.tsx`'s own gradient position.

**Three foil layers, not one — a single reactive gradient reads as "a stripe sliding," not "shimmering".** After the rotation-reactive fix above, the foil still didn't feel "alive" on a real phone. Real foil (and the standard CSS holo-card techniques it's modeled on) gets its liveliness from multiple things moving *differently* at once, not one thing moving correctly, so the sheen is now three stacked layers, all reading the same `foilTilt` prop but at different sensitivities/axes and different blend modes:
1. **Rainbow sweep** (`mix-blend-overlay`, the original layer) — a broad diagonal `linear-gradient`, `backgroundPosition` scaled by `foilTilt * 20`. Gives the overall color wash.
2. **Diffraction bands** (`mix-blend-overlay`) — two independent `repeating-linear-gradient`s at different angles (70deg/20deg), each tied to a *different* axis of `foilTilt` (steep stripes to `x`, shallow stripes to `y`) at different pixel-offset multipliers, so the two stripe patterns visibly cross and re-align with tilt instead of moving in lockstep — closer to how a real diffraction-grating hologram looks (fine bands separating/converging) than a single smooth gradient can get. **A first version used a ~20px period with fully opaque, hard-edged adjacent bands and larger position multipliers** — on a real device that read as harsh flickering "old TV static" rather than a soft hologram texture, since the pattern shifted by a full band-width within just a couple of pixels of drag. Widened to a 40-70px period with soft transparent-to-color fades (not hard edges) and roughly halved position multipliers to fix both the harshness and the flicker.
3. **Specular glare** (`mix-blend-screen`) — a small `radial-gradient` glint, `backgroundPosition` scaled by `foilTilt * 65`, much more sensitive than the other two layers.

`mix-blend-screen` (always brightens) is correct for the glint; it would wash out the habitat art the same way `color-dodge` did if used for the broader layers instead (see the overlay-vs-color-dodge note above) — layer 3 deliberately uses a different blend mode from 1 and 2 for this reason. All three layers share the same `foilTilt` prop and the same bottom mask-fade before the description text. The speed/axis differences between the three are what actually sell "alive" instead of "sliding" — a single layer, however well-tuned, reads as one thing moving correctly rather than a surface shimmering.

**`mix-blend-overlay`, not `color-dodge`** (tried first): color-dodge only ever brightens, which blew the effect out to a flat/washed look against bright habitat art — most visibly on mobile Safari's wide-gamut rendering, which runs hotter than desktop Chrome for the same blend math. `overlay` respects the underlying pixel's own luminance (darkens over dark art, lightens over light art) so it reads as a sheen rather than a haze. The default `background-position` is also offset toward the art's top-left (not centered) so the diagonal rainbow band (gradient angle 115deg, roughly top-left-to-bottom-right) is visible at a glance at rest, without requiring the user to tilt the card first to discover it — gradient stop alphas and the layer's overall opacity were both turned down at the same time, since the first version was "too strong" by default independent of the blend-mode choice.

This is the third approach tried, after two others both failed for reasons tied to the 3D transform tree: (1) animating `box-shadow` on `flipperRef` (`.card-flipper`, inside `.card-perspective`) — travels with the card during rotation, but reintroduces the iOS habitat-flash-during-drag compositing bug, because continuously mutating *any* visual property (not just CSS `animation`) on an element inside the 3D context triggers it; (2) animating `filter` on `innerRef` (`.card-scene-wrapper`, an ancestor of `.card-perspective`) — avoids the compositing bug, but silently renders nothing, because WebKit/Chromium don't reliably rasterize `filter` applied to an ancestor of a `preserve-3d` subtree. The `<canvas>` overlay shares no rendering context with the 3D tree at all, so neither failure mode applies. Trade-off (also true of attempt 2): the glow doesn't visually rotate with the card mid-flip, since it's anchored to the non-rotating outer wrapper. `.card-glow-wrapper` in `index.css` is a shape-anchor-only class (border-radius, no animation) kept for reference but is not applied to any element. **CSS `animation` must never be used for this glow** regardless of which element hosts it, for the same GPU-layer-promotion reason as above.

`CardScene.clampTilt` is exported and unit-tested in isolation (pure function, no DOM needed for the math itself).

**Mobile responsiveness** (`CardScene.tsx`): `Card.tsx` stays fixed at a 420px design width internally — its 3-layer px-based stacking (see above) isn't worth making fluid. Instead `CardScene` wraps it in a `ResizeObserver`-driven container: an outer measuring `<div>` (`outerRef`) reports the available width, `computeFitScale(availableWidth, 420)` (pure function, unit-tested) computes a `0–1` scale factor, and an inner `<div>` (`innerRef`) applies `transform: scale(...)` with `transformOrigin: 'top center'`. Because the scale is a single CSS transform on the wrapper, all three of `Card.tsx`'s absolutely-positioned sibling layers (frame/dino/text) shrink together and never drift out of alignment. **The fit-scale is width-only** — an attempt to also fit to viewport height (to avoid scrolling to reach the buttons below) was reverted after it shrunk the card far more than necessary on both mobile *and* desktop; don't reintroduce a height constraint here without explicitly gating it to narrow/mobile viewports, and expect users to scroll a little to reach the buttons on short screens. The outer wrapper's `height` is set explicitly to `(naturalHeight + DINO_OVERFLOW_BUFFER_PX) * scale` (buffer `56`px, for the dino's pop-out past the frame edges) since `transform` doesn't affect layout box size — without that, the wrapper would reserve the wrong height.

**The outer/inner wrapper divs must not be `display: flex` with default `align-items: stretch`** — an earlier version used `className="w-full flex justify-center"` on the outer div to center the inner div horizontally, but stretch silently compressed the *unscaled* inner div down to whatever height was last set on the outer div (itself derived from the *previous* measurement), creating a shrink-to-fit box far shorter than the card's real content height; the card then visually overflowed past that box (CSS transform doesn't get clipped by a box too small for its pre-transform content) and collided with the buttons below — this was the literal cause of a "card overlaps the share/download buttons on mobile" bug.

**The inner div must not be centered with `margin: auto` either** — that was the *next* fix attempted (`mx-auto` on a plain block instead of flex), and it's *also* wrong: `margin: auto` resolves to `0` (left-aligning, not centering) once the child's own intrinsic width (`420px`, unscaled — CSS transforms don't change the layout box used for margin resolution) exceeds the parent's width, which is exactly the case on any phone narrower than 420px. The card rendered visibly off-center as a result. The actual fix: `position: absolute; left: 50%` on the inner div with `transform: translateX(-50%) scale(${scale})` — `translateX(-50%)` centers using the element's own box regardless of how that compares to the parent's width.

**The fit-scale is measured in `useLayoutEffect`, not `useEffect`** — using `useEffect` let the card paint once at its full unscaled 420px size before the effect ran and corrected it, a visible flash/overflow on the first frame that `useLayoutEffect` (which runs before the browser paints) eliminates.

`CardScene` also has `onTouchMove`/`onTouchEnd`/`onTouchCancel` handlers mirroring the existing mouse-tilt handlers, so the tilt effect works on touch devices. Verified manually via Playwright at 320/375/390/667/844px combinations — no overflow, no button overlap, text legible, layout intact down to the narrowest common phone width (375×667, iPhone SE). The flip-corruption and centering fixes were specifically verified against real-device screenshots the user provided (iOS Safari); Playwright/Chromium did not reproduce either bug, so don't trust a clean Playwright run alone as proof these are fixed if either is touched again.

**Drag-to-rotate + drag-to-tilt, pivoting on the touch point**: dragging the card (`onTouchStart`/`onTouchMove`/`onTouchEnd` on `.card-perspective`) rotates it live with the finger **horizontally** — `spinDeg` (rotateY) is set directly from the horizontal drag delta (`DRAG_SENSITIVITY_DEG_PER_PX = 0.6`) with `transition: 'none'` so the rendered rotation always matches the touch position exactly, no easing lag. On release it snaps to whichever face (front/back, nearest multiple of 180°) ended up closer, animated over `DRAG_SNAP_TRANSITION`. Because the rotation is driven directly by state during a drag (no CSS-interpolated unknown), `facing` is computed synchronously from the live angle on every `touchmove` rather than via the `setTimeout` approximation used for the button-triggered spin/flip-in (where the actual visual angle is CSS-interpolated and JS can't read it without polling — the timer approach there is a deliberate, separate technique, not a leftover to consolidate).

**Vertical drag drives a clamped tilt** (`tilt.rotateX`, ±`MAX_TILT_DEG` = 15°, same axis/state the desktop hover effect already uses) instead of doing nothing or nudging position — two earlier designs were tried and superseded here on explicit request: page-scroll passthrough (canceling the card drag on a vertical gesture so the page could scroll normally), then a small clamped `translate(x, y)` position nudge. Neither gave a convincing "3D" feel on a real phone.

**The pivot stays at the card's default center — anchoring it to the touch point was tried and reverted.** An earlier version set `transformOrigin` on `.card-flipper` to wherever the touch started, hoping to make dragging feel like grabbing the card at that spot instead of spinning it uniformly around its center. On a real device this looked broken rather than "grabbed": an off-center pivot combined with 3D `perspective` visibly translates the whole element toward whichever edge was touched, worse near the corners. Reverted on explicit feedback; the horizontal-rotation + vertical-tilt combination above is what's carrying the "3D" feel now, not the pivot point.

Because touching the card now always drives the card rather than sometimes falling through to the page, `.card-perspective` sets `touch-action: none` (overriding the `touch-action: manipulation` inherited from `html`/`body`) so the browser doesn't also try to pan the page during the same gesture; touches starting outside the card are unaffected and scroll normally.

**The button-triggered spin's facing/reset logic must be relative to the spin's *actual* starting face, never hardcoded** — since drag-to-rotate can leave the card resting on its back, and "Girar carta" is a flat `spinDeg += 360` (always ends visually where it started), an earlier version unconditionally reset `facing` to `'front'` and `spinDeg` to `0` when the spin's `setTimeout` completed. If the spin was triggered while showing the back face, that hardcoded reset snapped the card to the front face instantly right as the animation finished — a visible jump-cut ("the card disappears for a frame"). Fixed by capturing `startFacing` before the spin begins, sequencing the mid-spin facing toggle relative to it (`front→back→front` or `back→front→back`), and resetting `spinDeg` by literally subtracting the `360` that was added (`current - 360`) rather than to a hardcoded `0`, so the end state is guaranteed identical to the start state.

**Zoom is locked app-wide** (`index.html`'s viewport meta: `maximum-scale=1.0, user-scalable=no`; `touch-action: manipulation` on `html, body` in `index.css`) since this is a kids' app and the only gesture that should do anything is the card's own tilt/flip — pinch-zoom and double-tap-zoom are disabled everywhere, not just the result screen (no per-route way to scope a `<meta viewport>` tag). Note iOS Safari has ignored `user-scalable=no` for accessibility since iOS 10; `maximum-scale=1.0` plus `touch-action: manipulation` is the practical mitigation, not a hard guarantee.

**`background-color` must be set on `html`, not just `body`** — iOS Safari's rubber-band overscroll bounce reveals whatever is behind `body` when you scroll past the content edge, which is `html`'s background; left at the browser default (white), this caused a visible white flash at the top/bottom of the page during normal scrolling. Fixed by setting the same `#0d1a0f` on both `html` and `body` in `index.css`, plus `overscroll-behavior-y: none` to disable the bounce outright.

### Known gaps / defects

- **On short screens, a touch starting on the card can no longer scroll the page.** `.card-perspective` now sets `touch-action: none` so drag-to-rotate/tilt fully owns touches on the card (see "Drag-to-rotate + drag-to-tilt" above). Before this, a vertical drag starting on the card fell through to native page scroll; now it always moves the card instead. If the "scroll a little to reach the buttons below" trade-off documented in Mobile responsiveness above becomes a real problem, a user on a short screen who tries to scroll by touching the card itself will find it doesn't work — they'd need to scroll from the background instead. Unverified on a real device; worth checking if this comes up.
- **The user has repeatedly reported a visual fix as "not visible" when it was actually live — twice now, both confirmed via `curl`-ing the deployed JS bundle and grepping for the relevant class/string.** Deploys here are via GitHub Actions (`.github/workflows/deploy.yml`, triggers on every push to `main`, runs `npm run build` + `wrangler pages deploy`) and have been consistently fast (~40s) and successful (check `gh run list --workflow=deploy.yml`) — the gap has been the user's browser/CDN cache, not a missing deploy. If a reported-as-broken fix was definitely committed, pushed, and deployed (verify all three before assuming the fix itself is wrong), say so explicitly and suggest a hard refresh / incognito tab before re-investigating the code.
- **html2canvas download doesn't see the dino's 3D pop-out, and also doesn't see the mobile fit-scale wrapper.** The hidden capture node in `App.tsx` (`<Card ref={certificateRef} .../>`, used by `captureCertificateAsPng`) renders the bare `Card` component directly, not through `CardScene`'s scaling wrapper — this is intentional (the download should always be the full-resolution 420px card, not whatever scale the viewport happened to render at), but it also means if a dino's cutout extends past the framed box's edges, html2canvas — which canvases based on the target ref's own bounding box — will likely clip it at that boundary in the downloaded PNG, even though it's visible on screen. Not yet verified against a real (non-placeholder) generated image; worth checking before relying on the pop-out effect surviving the download.
- **`docs/RARITY_SYSTEM.md`'s worked examples contain real arithmetic errors** (caught during implementation — e.g. a fixture the doc calls "Épico" actually scores "Raro" under its own point table). Trust the point tables over the doc's narrative examples; `speciesHash.test.ts` and `Card.test.tsx` have inline comments where this was corrected.
- **`ATTRIBUTE_SLUGS`/file-naming mismatches are intentional, not bugs**: project strings use accented Spanish (`'Súper garras'`, `'Escamas coloridas'`, `'Ártico'`) while `docs/RARITY_SYSTEM.md`'s reference implementation used slightly different un-accented keys (`'Super garras'`, `'Escamas'`) — `speciesHash.ts`'s point tables were re-keyed to match the project's actual strings, not the doc verbatim.
- **`dinoCutout.ts` requires same-origin images** (canvas `getImageData` throws/taints on cross-origin without proper CORS headers). Production images are served same-origin via `/images/[key].ts`, so this is fine in practice, but don't point `result.imageUrl` at a third-party host without revisiting this.
- **The wizard's expanded attribute lists (6/4/6/4/6 = 3,456 combos) are validated independently in two places**: `src/data/attributes.ts` (frontend option lists) and `functions/api/generate-dino.ts`'s `VALID_SIZES`/`VALID_HABITATS`/`VALID_DIETS` (backend request validation). They must be kept in sync manually — there's no shared single source of truth for the *option lists* (there is one for the *types*, in `shared/types.ts`).
- No automated browser/visual tests exist for the card's CSS layout (border thickness, overlap, drop-shadow positioning) — verification during development has been manual (Playwright MCP screenshots in dev sessions, not committed as tests). Layout regressions in `Card.tsx` won't be caught by `npm test`.
- **`certificate.ts`'s download/share flow has never been verified on a real iOS device and is a plausible silent-failure point.** `navigator.share()` is called only after several `await`s (`html2canvas()` render, `canvas.toBlob()`, and in the email-gate path also a prior `await subscribeEmail(...)` network call) — iOS Safari requires `share()` to fire within the synchronous tail of the originating user gesture, and this almost certainly doesn't qualify, meaning `share()` likely throws and falls through to the `<a download>` blob-URL fallback, which has its own history of inconsistent support on iOS Safari. This is unverified, not confirmed broken — but it's the single highest-impact untested risk in the app, since download/share is the entire payoff moment. If you're asked to fix it, the real fix requires restructuring so `share()` (or the download click) happens synchronously in the click handler with a pre-existing blob, not after async work.
- **No automated a11y or visual-regression tooling** (`package.json` has no `jest-axe`-equivalent, no Percy/Chromatic-equivalent) — every mobile-layout fix in this project to date has relied on manual Playwright screenshots taken during the session plus the user checking a real device; nothing in `npm test` would catch a regression to `Card.tsx`/`CardScene.tsx` layout or accessibility.

### Testing conventions

Vitest + Testing Library, jsdom environment (`src/test-setup.ts`, which also stubs `ResizeObserver` since jsdom doesn't implement it — needed by `CardScene`'s fit-to-container scaling). One test file per component/util, colocated (`Foo.tsx` + `Foo.test.tsx`). Backend functions tests live under `functions/**/*.test.ts` and fake the KV/R2 bindings with plain in-memory objects (see `createFakeKV`/`createEnv` helpers repeated per test file — no shared mock library). Cloudflare-specific globals (`R2Bucket`, `PagesFunction`) come from `@cloudflare/workers-types`.
